# 「拾光」截图记忆管家 Agent — 设计文档

- 日期:2026-07-20
- 赛事:中国高校计算机大赛 · 人工智能创意赛 · AIAgent 创新赛题
- 状态:总体设计已确认，实施进行中（Story 2 Task 2.5 待开始）

## 0. 一句话创意(提交项 1)

拾光是一个"长"在鸿蒙系统里的截图记忆管家 Agent —— 它主动理解你随手截下的每一张图(海报、聊天记录、课表、缴费通知),在对的时间、对的地点把里面藏着的事情推到你面前,让截图从"存了就忘的坟场"变成"会自己开口提醒你的记忆"。

## 1. 产品定位与创意背景

### 1.1 第一性原理推导

1. **人为什么截图?** 截图是成本最低的"稍后处理"动作:看到讲座海报、老师发的 DDL、取件码,来不及处理就先截下来。截图的本质是一条待办 + 一份记忆,但它以像素形式存在,系统不理解它。
2. **痛点在哪?** 截图存下的那一刻,信息就死了。相册里平均堆积数百张截图,90% 再也没被打开过。讲座错过、优惠券过期、取件码找不到 —— 不是没记,是记了找不回、想不起。
3. **为什么必须是 Agent 而不是搜索功能?** 搜索要求用户"记得自己截过";真正的痛点是用户忘了它存在。核心能力不是"找得到",而是"它主动找你"。
4. **为什么必须是鸿蒙?** 截图监听、日程写入、元服务卡片、通知时机都是系统级能力。拾光天然是"系统伙伴"而非需要打开的 App,这是"鸿蒙技术创新性"的直接答题点。

### 1.2 与赛题主题的对应

| 赛题主题 | 拾光的回答 |
|---|---|
| 个人日常生活与陪伴 | 长期个人记忆,回答带建议("还有 5 天过期,要现在用吗?") |
| 主动服务 | 时间/地点/对话三类触发,在对的时刻主动浮现 |
| 自然交互 | 视觉理解截图 + 自然语言召回 + 系统级卡片交互 |

### 1.3 演示三主线(评审视角的作品)

1. 截一张讲座海报 → 静默解析 → 讲座前 1 小时卡片主动弹出"你上周截过这个讲座,在 XX 楼,要出发了" → 一键导航。
2. 截一张取件码 → 接近快递站 → 取件码自动浮现在卡片,一键复制。
3. 随口问"我上次截的那个奖学金申请截止是哪天?" → 自然语言召回原图 + 一键写入日程。

演示纪律:在"主动服务"演示段落中,全程不得出现"打开 App 手动操作"的画面。

## 2. 核心功能与交互设计

围绕"截图的一生"分四个模块:

### ① 静默摄取(Ingest)

- 用户正常截图,无需打开拾光。通过 `photoAccessHelper` 媒体库变更通知监听新增截图,后台自动拉取。
- 每张截图执行:端侧 OCR 提取文字 + 多模态大模型语义理解,输出结构化卡片:`{类型: 讲座海报/取件码/缴费通知/聊天记录/课表…, 关键实体: 时间/地点/码号/金额, 可执行动作: 提醒/导航/复制码}`。
- 交互克制原则:解析完成仅在通知栏轻提示一次"已读懂 1 张截图",点开可纠错。不打扰是主动服务的前提。

### ② 时机引擎(Trigger)

每个解析出的事件绑定触发条件,三类:

- **时间触发**:讲座前 1 小时、优惠券过期前 1 天、DDL 前 3 天(后台任务 + 通知)。
- **地点触发**:接近快递站浮出取件码、接近教学楼浮出当日课表(地理围栏)。
- **对话触发**:用户问到相关话题时召回(见 ③)。

推送形态使用鸿蒙实况窗/通知卡片,卡片直接带动作按钮:「复制取件码」「导航过去」「写入日程」,从提醒到行动零跳转。

### ③ 自然语言召回(Recall)

- App 内对话框 + 时间轴。支持模糊问法:"上个月截的那个健身房优惠还在吗""帮我找有二维码的那张"。
- 检索:结构化字段过滤 + LLM 语义挑选,回答附原图,且带记忆 + 建议("你 3 月 2 日截的,还有 5 天过期,要现在用吗?")—— 对应"陪伴"维度。

### ④ 元服务卡片(Widget)

- 桌面卡片常驻"今日相关":今日 DDL、待取快递、当天课表截图自动轮换。演示视频中最直观的鸿蒙创新画面。

交互全景:截图(无感)→ 通知轻确认(可纠错)→ 卡片主动出现(对的时刻)→ 对话召回(想不起来时)。

## 3. 技术架构与实现路径

总体结构:**端侧为主 + 云端大模型 API,不自建后端**。少一个部署环节,隐私叙事更好("截图不离开手机,只上传脱敏文本 + 低分辨率缩略图")。

```
┌─ HarmonyOS NEXT App (ArkTS) ─────────────────────────┐
│  摄取层   photoAccessHelper 监听新截图                  │
│    ↓     Core Vision Kit OCR(端侧、免费、离线)          │
│  理解层   OCR文本 + 缩略图 → 云端多模态 LLM API          │
│    ↓     → 结构化 JSON{类型/实体/触发条件/动作}          │
│  记忆层   RelationalStore(本地 SQLite)存卡片            │
│  触发层   backgroundTaskManager + reminderAgent(时间)   │
│          Location Kit 地理围栏(地点)                    │
│  呈现层   ArkUI 时间轴&对话页 / Form Kit 元服务卡片       │
│          / 通知栏动作按钮                               │
└──────────────────────────────────────────────────────┘
```

### 关键选型

1. **OCR 用鸿蒙 Core Vision Kit**:端侧、免费、离线可用;使用鸿蒙 AI 能力本身即鸿蒙创新分。LLM 只接收 OCR 文本 + 低分辨率缩略图。
2. **理解层按 OpenAI-compatible 协议对接,厂商可由用户配置,但要求必须是多模态模型**:统一走 OpenAI Chat Completions(vision 消息格式)调用,`baseURL / apiKey / model` 三项在设置页可改,任何 OpenAI-compatible 多模态模型(智谱 GLM-4V、通义 Qwen-VL/Plus、豆包 Seed vision、GPT、Claude 等)即插即用。**不内置任何默认 key**(打进安装包的 key 可被提取,有泄漏与盗刷风险):首次启动强制进入配置向导,填写并通过验证门禁后才能进入主界面;配置页提供常见厂商 baseURL 快捷模板(智谱/通义/豆包/OpenAI)减少填错。演示视频中配置向导本身作为开场画面展示,顺便体现开放性。Prompt 固定输出 JSON schema;解析失败自动重试一次,再失败落为"未分类截图",可人工补标。
   **保存时验证门禁**:用户保存设置时,后台发送一次带可校验答案的小型颜色/形状关系挑战图的探测请求。成功且答案匹配才允许保存;若返回"不支持图片内容"类错误或答案不匹配,拒绝保存并提示"该模型不支持图像输入,请填写多模态模型(如 glm-4v、qwen-vl-plus)"。填错当场拦截,而不是等截图全部解析失败后才暴露。不做纯文本降级模式——各厂商均有多模态型号,限制对用户成本极低,而双模式管线会显著增加实现与测试复杂度。
   **OCR 与多模态的关系**:端侧 OCR(Core Vision Kit)的文本与缩略图一起送入模型,目的不是备胎而是数字精度——取件码、金额、截止日期这类内容,模型直接读图偶尔错认个别字符,OCR 文本作为对照可显著压低此类错误;OCR 文本同时支撑本地关键词检索与离线排队。
3. **不做向量库**:结构化字段过滤 + 候选卡片摘要进 LLM 上下文做语义挑选。个人截图量级(几百张)足够,省约一周工作量。
4. **触发层为唯一真机风险点**:后台任务与地理围栏在模拟器上不可靠。兜底:时间触发用调系统时间实拍;地理围栏受限时降级为"打开 App 时按当前位置排序卡片"。
5. **元服务卡片独立开发、最后集成**:只读 RelationalStore 的"今日相关"视图,与主 App 解耦,做不完不影响主线。

### 错误处理原则

任何一层失败都降级不阻断:OCR 失败 → 存原图待重试;LLM 失败 → 未分类;无网络 → 摄取照常、理解排队。

LLM 调用失败按错误类别分流,而不是一律重试:

| 错误类别 | 处理 |
|---|---|
| 认证失败(401/403) | 停止排队,通知栏提示"请检查 API Key",引导去设置页 |
| 不支持图片内容 | 正常情况下已被设置页验证门禁拦截;运行期意外出现(如厂商改接口)则暂停解析队列,通知栏提示"当前模型不再支持图像,请更换模型" |
| 限流/超时(429/5xx) | 指数退避排队重试,不丢任务 |
| 返回非法 JSON | 重试一次,再失败落"未分类" |

### 分工(3-4 人,前端/TS 背景)

- A:ArkUI 界面 + 元服务卡片
- B:摄取/存储/触发等系统能力
- C:LLM prompt + 解析管线
- D:文档 + 演示视频 + 测试

## 4. MVP 边界

| 做(演示主线必需) | 不做(文档中写为"规划中") |
|---|---|
| 截图监听 + OCR + LLM 结构化 | 相册全量历史扫描(只处理安装后新截图) |
| 4 类截图:海报/取件码/DDL 通知/优惠券 | 开放全品类理解(不认识归"未分类") |
| 时间触发 + 地点触发(含降级方案) | 对话历史、位置轨迹深度上下文建模 |
| 对话式召回 + 时间轴 | 语音交互(演示用系统输入法语音替代) |
| 元服务卡片"今日相关" | 多设备流转、手表端 |
| 首次启动模型配置向导(baseURL/apiKey/model,验证通过才可进入) | 多模型自动路由、本地端侧大模型、内置默认 key |

裁剪逻辑:宁可 4 类截图丝滑跑通,不要 20 类半残 —— 完备度 20 分考"体验完整顺畅",不是功能数量。

## 5. 市场前景

- **需求真实性**:截图是手机上频次最高的非正式记录行为,人人相册里有数百张僵尸截图,评审可直接共情。
- **社会价值**:信息过载时代的个人记忆基础设施;对老年人(缴费通知、健康提示类截图)有天然适老延展。
- **商业路径**:免费个人版 → 会员(云备份/全品类理解);对华为的价值是元服务生态示范应用,有被系统能力吸纳的想象空间。
