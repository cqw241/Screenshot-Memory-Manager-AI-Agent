# 拾光 Story 3：能力分级的截图自动摄取设计

- 日期：2026-07-22
- 状态：设计已确认，待实施
- 分支：`feature/s03-silent-ingest`
- 上游基线：Story 2 已通过 PR #2 合并；真机 OCR 仍是补验债务，但不阻塞 Story 3 的接口与队列开发
- 对应总计划：`docs/DevelopmentPlan.md`

## 1. 文档目的

Story 3 的原计划把“普通三方应用在用户不打开 App 时监听任意截图，并在数秒内自动解析和通知”视为确定能力。代码、SDK 与平台规则核对后，该前提不能直接成立：媒体库全量读取依赖受限权限，媒体变更回调也不等于系统会在进程不存在时唤醒应用。

本设计采用“先真机验证、再按能力分流”的方案。Story 3 不以伪造的后台能力换取表面验收；探针失败时允许按真实平台能力降级并完成 Story，产品和文档必须准确显示实际等级。

## 2. 已确认的核心决策

1. 新增 Task 3.0 平台能力探针，先验证权限、截图来源识别、前后台回调和通知，再决定生产路径。
2. 采用“持久化对账优先”，媒体变更监听只作为立即对账的提示，不作为唯一事实来源。
3. 能力分为 L0、L1、L2，Story 按最终等级执行对应验收，不再用固定的后台承诺覆盖所有设备与签名条件。
4. 自动识别默认关闭，必须由用户在前台理解数据流并主动开启。
5. Story 3 建立最小可恢复队列；指数退避、联网自动恢复和完整错误分流留给 Story 8。
6. “杀进程续跑”解释为应用下次获得运行机会后恢复，不承诺强制停止后仍被系统唤醒。
7. 纠错只修改当前卡片类型，并重新派生建议动作；不实现从用户纠错中自动学习 Prompt。

## 3. 当前基线与约束

### 3.1 可复用的 Story 2 能力

- `ScreenshotIngestionCoordinator` 已串联单图导入、OCR、模型理解与卡片写入。
- `ScreenshotIngestionPorts` 已提供理解器和卡片写入器窄接口。
- `ScreenshotImportService` 已实现 PhotoPicker 选图、私有复制与 OCR，但入口依赖用户主动选图。
- `ScreenshotCardRepository` 已支持创建、查询、更新理解结果、删除和时间轴查询。
- `ScreenshotCardRepository.updateUnderstanding()` 可作为纠错写入入口。
- Story 2 已验证模拟器上的 OCR 空结果降级、多模态理解、非法 JSON fallback、持久化和时间轴展示。

### 3.2 尚不存在的能力

- 已知媒体 URI 的自动导入入口；
- 媒体库变化适配器与截图来源判断；
- 增量扫描游标和遗漏补偿；
- 持久化解析队列；
- 自动卡片的来源幂等键；
- 完成通知、深链和纠错页；
- 相册与通知权限的产品化启用流程。

### 3.3 平台边界

- `photoAccessHelper` 的媒体变化监听和媒体读取依赖 `ohos.permission.READ_IMAGEVIDEO`；本地 API 24 SDK 将该权限标记为 `system_basic`、`user_grant`。
- 受限权限能否用于本项目的可交付签名，必须以实际申请与真机安装结果为准。修改本地签名模板或提升 APL 只能证明实验能力，不能关闭正式验收项。
- 注册媒体回调只能证明当前进程存活时可以接收事件，不能证明进程被回收或强制停止后会被系统启动。
- HarmonyOS 持续任务只允许匹配规定场景。被动监听截图不能借用无关后台模式，且持续任务本身会产生用户可感知通知，不符合“静默”承诺。
- Story 3 继续以 API 24 编译、API 20 目标与兼容版本为基线。若探针发现只有 API 21 以上专属接口才能可靠实现，必须另行评审兼容策略，不能直接引入。

## 4. 原计划的不足与矛盾

| 原计划 | 问题 | 本设计修正 |
|---|---|---|
| 首次触发时申请相册与通知权限 | 没有相册权限时无法先收到媒体触发；后台也不应弹授权框 | 用户在前台主动开启并依次申请权限 |
| 媒体新增事件直接入队 | 进程挂起、回调重复、乱序或服务重启时会漏图或重复 | 监听只发提示，统一执行带游标和重叠窗口的增量对账 |
| 按相册分类或文件名筛选截图 | API 20 可用字段与不同设备行为未验证；文件名可能本地化或变化 | Task 3.0 先记录真机字段，只接受有证据的高置信规则 |
| 杀掉 App，重启后队列续跑 | “被系统回收”“用户强制停止”“下次启动恢复”含义混在一起 | 明确为下次获得运行机会后恢复；强制停止不承诺自动唤醒 |
| Story 3 建持久化队列，Story 8 再建持久化队列 | 重复建设且职责不清 | Story 3 建基础持久队列，Story 8 强化错误分类、调度与退避 |
| 修改类型后“同步后续触发” | 容易被理解为在线学习或自动改 Prompt | 仅保证未来触发读取更新后的同一卡片数据 |
| 每张图一条通知且“不重复轰炸” | 连续截图时表述含糊 | 每卡最多一条，允许系统归组；通知权限关闭时降级为应用内未读状态 |
| 固定要求后台数秒完成 | 普通发布权限与后台生命周期可能无法满足 | 使用 L0/L1/L2 分级验收并准确展示能力 |

## 5. 目标与非目标

### 5.1 目标

- 用真机证据确定当前签名和设备可达到的自动摄取等级。
- 在 L1/L2 下可靠发现安装并启用功能之后的新截图，不处理历史全量图库。
- 用持久化、串行、幂等队列复用 Story 2 主管线。
- 成功后投递可深链到具体卡片的轻通知，并允许修改类型。
- 权限拒绝、撤回、进程退出和重复媒体事件均不丢任务、不重复建卡、不崩溃。
- 对外描述与实际能力一致。

### 5.2 非目标

- 绕过系统权限、使用非本业务类型的持续任务或修改签名模板冒充发布能力；
- 扫描安装或启用前的全部历史截图；
- 强制停止应用后仍保证自动运行；
- Story 8 的指数退避、联网监听、定时后台重试和完整熔断；
- Story 9 的 Prompt 精度调优；
- 从纠错行为中自动训练模型或改写 Prompt；
- 删除相册源图或自动清理 Story 2 已保存的私有图片。

## 6. 方案比较

### 6.1 监听器优先

媒体新增事件到达后立即把 URI 入队。实现最短，但事件遗漏、重复、乱序和进程恢复都要单独打补丁，不适合作为可靠记忆入口。

### 6.2 持久化对账优先（采用）

启动、回前台和媒体事件都调用同一个 `ScreenshotReconciler`。对账器从持久游标之前的一小段重叠时间窗重新查询候选，依靠稳定来源键去重。事件丢失不会永久丢图，事件重复也不会重复建卡。

代价是需要明确游标、来源键、查询范围和数据库迁移，但这些复杂度集中在一个模块内，并直接支撑 Story 8。

### 6.3 定时轮询

周期性扫描相册。逻辑直观，但仍受后台调度限制，额外耗电且不符合静默工具的克制原则，因此不采用。

## 7. 能力等级与决策门禁

| 等级 | 判定条件 | 产品行为 |
|---|---|---|
| L2 静默摄取 | 可交付签名可获相册读取权限；前台与后台驻留回调稳定；截图可高置信识别 | 前后台驻留时事件触发立即对账；启动和回前台继续补扫 |
| L1 前台与回前台补扫 | 可获相册读取权限，但后台驻留回调不稳定或不可用 | 前台监听；启动和回前台增量补扫；不宣称后台实时通知 |
| L0 手动导入 | 可交付签名无法获得全量相册读取权限，或无法可靠识别截图来源 | 保留 PhotoPicker 手动导入和类型纠错；不注册自动监听，不展示虚假能力 |

Task 3.0 必须分别记录运行时能力和分发资格。只有正常可交付签名的结果可以决定最终等级。特殊本地签名结果可作为附录，但不能把 L0/L1 提升为 L2。

## 8. 总体架构

```text
启动 / 回前台 / 媒体变更回调
              ↓
      ScreenshotReconciler
  查询游标后的候选，并保留重叠时间窗
              ↓
      ScreenshotClassifier
  结合相册归属、媒体类型、时间和名称证据
              ↓
        DurableIngestQueue
   以 sourceMediaKey 幂等入队并串行消费
              ↓
 KnownUriImporter → OCR → S2 理解 → 卡片落库
              ↓
      CompletionNotifier
              ↓
      cardId 深链纠错页
```

### 8.1 模块职责

#### CapabilityProbe

Task 3.0 专用探针。采集权限申请、媒体字段、前后台回调、通知与签名证据，输出 L0/L1/L2 结论。它不承载生产业务逻辑。

#### MediaChangeSource

封装监听注册、注销与回调，只发出“媒体可能变化”的信号。不得持有页面状态、模型配置、API Key、队列表或卡片仓库。

#### ScreenshotReconciler

维护扫描游标和重叠窗口，查询安装且启用功能之后的候选媒体。多个触发并发到达时合并为一次当前对账加一次尾随对账。

#### ScreenshotClassifier

根据 Task 3.0 证明可靠的字段判断来源。强证据优先于文件名；无法高置信确认时宁可忽略，不把拍照或普通保存图片上传给模型。

#### DurableIngestQueue

负责唯一入队、任务状态、串行 claim、崩溃恢复和暂停。它不负责指数退避或网络状态调度。

#### KnownUriImporter

把已知媒体 URI 复制到应用私有目录并执行 OCR，返回与 Story 2 相同的 `ReadyImportOutcome` 语义。不得伪造 PhotoPicker 点击流程。

#### AutoIngestionRunner

消费队列并复用 Story 2 的 `ScreenshotUnderstander` 与卡片写入能力。页面只订阅结果，不参与执行队列。

#### CompletionNotifier

根据任务 ID 生成稳定通知 ID，以 `cardId` 构造 WantAgent 深链。通知权限关闭时记录未通知原因并结束任务。

#### CorrectionPage

读取 `cardId` 对应卡片，允许修改类型。保存时保留实体字段，根据新类型与实体重新派生建议动作，并调用卡片仓库统一更新。

## 9. 数据设计与一致性

### 9.1 自动来源键

- 自动发现的每个媒体项生成稳定 `sourceMediaKey`。
- 优先使用平台稳定媒体标识；若探针证明 URI 在目标设备上稳定，可使用规范化 URI；否则使用经验证的标识组合。
- 队列表对 `sourceMediaKey` 建唯一约束。
- 卡片增加可空且唯一的 `sourceMediaKey`：自动卡片必填，Story 2 手动导入保持为空。
- 队列重跑前先查已有自动卡片，数据库唯一约束作为最后防线。

### 9.2 最小任务字段

- `taskId`
- `sourceMediaKey`
- `sourceUri`
- `discoveredAt`
- `state`
- `attemptCount`
- `privateImageDescriptor`（复制完成后可用，包含恢复所需的私有 URI、类型、尺寸和字节数）
- `cardId`（卡片落库后可用）
- `lastErrorCategory`
- `lastErrorAt`
- `notificationStatus`
- `updatedAt`

字段只保存恢复所需信息，不保存 API Key、OCR 全文、模型原始响应或用户图片内容副本。

私有复制必须对同一任务幂等：使用任务稳定标识定位私有目标，重复执行时复用完整且可读的既有副本；发现半写入或损坏文件时先安全重建，再推进任务状态。这样进程在“文件已复制、状态尚未提交”的窗口退出，也不会无限制造私有副本。

### 9.3 队列状态机

```text
PENDING → PROCESSING → CARD_SAVED → COMPLETE
             │
             ├→ PENDING
             ├→ PAUSED_PERMISSION
             ├→ PAUSED_CONFIG
             └→ FAILED_TERMINAL
```

- 遗留 `PROCESSING` 在下次启动时恢复为 `PENDING`。
- `CARD_SAVED` 恢复时只结算通知，不再次调用 OCR、模型或建卡。
- 权限恢复后，`PAUSED_PERMISSION` 回到 `PENDING`。
- 有效模型配置保存后，`PAUSED_CONFIG` 回到 `PENDING`。
- 源图在复制前已不存在或永久不可读时进入 `FAILED_TERMINAL`。
- 通知 ID 由任务 ID 稳定派生；重复发布覆盖同一通知。

## 10. 数据流

### 10.1 发现与对账

1. 启动、回前台或媒体回调请求对账。
2. 对账器读取持久游标，并从游标前的重叠时间窗查询新增候选。
3. 分类器只保留高置信截图。
4. 队列按 `sourceMediaKey` 幂等插入。
5. 本轮查询成功后推进游标；部分查询失败时不越过未确认范围。
6. 对账期间收到新信号时，在本轮结束后再执行一次尾随对账。

### 10.2 自动摄取

1. 单消费者 claim 最早的 `PENDING` 任务并标记 `PROCESSING`。
2. 已知 URI 导入器完成私有复制和 OCR。
3. 复用 Story 2 理解服务；OCR 为空或异常仍可走图片理解。
4. 复用卡片仓库创建带 `sourceMediaKey` 的卡片。
5. 保存 `cardId` 并进入 `CARD_SAVED`。
6. 有通知权限时发布稳定通知；无权限时记录应用内未读状态。
7. 进入 `COMPLETE` 并继续下一张。

### 10.3 纠错

1. 通知 WantAgent 携带唯一 `cardId`。
2. 页面读取最新卡片，不信任通知内缓存字段。
3. 用户选择新类型。
4. 保留当前实体，重新派生建议动作并更新卡片。
5. 时间轴、详情页以及后续 Story 的提醒查询读取同一更新记录。

## 11. 权限与用户体验

### 11.1 首次启用

自动识别默认关闭。用户点击开启后，先展示：

- 只处理启用后的新增截图；
- OCR 在端侧执行；
- OCR 文本和最长边不超过 1024 px 的缩略图会发送到用户配置的模型服务；
- 可随时关闭，关闭不会删除已有卡片。

用户确认后先申请相册读取权限，再独立申请通知权限。相册权限决定 L0/L1/L2 摄取能力；通知权限只决定完成提示，不得阻断卡片创建。

### 11.2 关闭与撤权

- 用户关闭自动识别：立即注销监听、停止发现新任务、暂停尚未开始的自动任务；关闭后不得启动新的模型请求。已经发出的网络请求尝试取消，若底层传输不支持取消则允许当前请求结算，但不得继续下一项。
- 相册权限撤回：停止监听与补扫，任务进入或保持 `PAUSED_PERMISSION`，首页显示恢复入口。
- 通知权限拒绝或撤回：继续建卡，使用首页未读状态；后续恢复权限时不补发历史通知。
- 已生成卡片始终保留，除非用户在详情页明确删除卡片记录。

### 11.3 能力状态文案

- L2：“自动识别已开启，可在应用后台驻留时处理新截图。”
- L1：“自动识别已开启，将在打开或返回拾光时检查新截图。”
- L0：“当前安装无法自动读取新截图，请使用手动导入。”

不得使用“始终后台运行”“强制停止后仍可处理”等无证据文案。

## 12. 通知与纠错规则

- 每张成功生成的自动卡片最多一条完成通知；系统可对连续通知归组。
- 已分类文案：“已读懂 1 张截图 · {类型}”。
- fallback 文案：“已保存 1 张截图，点此补充类型”。
- 网络、认证或存储失败不发布成功通知。
- 通知点击必须直达对应卡片纠错页，而不是只打开首页。
- 类型纠错只影响该卡片；不修改 OCR 文本、实体原文或其他卡片。
- 修改类型后重新派生动作，避免保留不适用于新类型的旧动作。

## 13. 错误处理与 Story 8 边界

### 13.1 Story 3 负责

- 队列持久化、串行执行和幂等；
- 进程退出后的状态恢复；
- 权限失效与模型配置失效的基础暂停；
- 源图永久不可读的终止；
- 可重试错误保留任务，并在下一次自然运行机会继续；
- 错误日志脱敏，不记录 URI 查询参数、API Key、OCR 全文或模型原始正文。

### 13.2 Story 8 负责

- 401/403、不支持图片、429、5xx、超时和网络断开的统一分类；
- 网络恢复感知；
- 30 秒、2 分钟、8 分钟等指数退避和封顶策略；
- 全局队列熔断与恢复；
- 认证和模型能力失效的专门通知与设置页深链。

Story 3 不承诺断网后在后台按定时器自动恢复，只保证任务不丢。

## 14. Task 执行计划

### Task 3.0 平台能力探针

#### 工作内容

1. 使用正常可交付签名，在目标真机申请并检查 `READ_IMAGEVIDEO`。
2. 分别在前台、后台驻留、系统回收后、用户强制停止后制造截图事件。
3. 截图、拍照、下载图片、普通保存图片各执行 3 次，记录回调类型、媒体类型、URI/标识、相册字段、名称和时间字段。
4. 验证通知申请、发布与 `cardId` WantAgent 深链。
5. 记录设备型号、HarmonyOS 版本、API、签名与 Profile 类型。
6. 输出 L0/L1/L2 结论和采用的截图判定规则。

#### 完成门禁

- [ ] 工作记录包含完整真机矩阵与原始脱敏结果
- [ ] 特殊本地签名与可交付签名结果分开记录
- [ ] 已明确最终能力等级和禁止使用的字段/后台假设
- [ ] 未取得真机或可交付签名证据时，不启动 L2 生产实现

### Task 3.1 自动摄取基础设施

L1/L2 执行。

#### 工作内容

1. 设计并迁移卡片 `sourceMediaKey`。

## 15. 测试策略

### 15.1 自动测试

- 能力等级判定：权限、监听和通知组合映射正确。
- 分类器：截图、拍照、下载、普通保存和证据不足用例。
- 对账器：空结果、边界时间、重叠窗口、乱序、重复、重新检查和查询失败。
- 队列：唯一入队、FIFO 串行、遗留 claim 恢复、暂停/恢复、终止和尾随运行。
- 一致性：建卡前退出、建卡后状态未更新、通知前退出、通知后状态未更新。
- 导入器：OCR 文本、空文本、异常和资源释放。
- 通知：稳定 ID、无权限、深链参数和重复发布。
- 纠错：类型修改、动作派生、卡片不存在和写入失败。
- 回归：Story 1 配置门禁与 Story 2 全部现有测试。

所有 OCR 内容断言使用确定性 fake，不用模拟器空 OCR 结果冒充真机质量证据。

### 15.2 设备测试

- 模拟器：页面、状态机、数据库迁移、队列、fake 媒体事件、通知路径与构建。
- 真机：受限权限、真实媒体字段、截图筛选、前后台回调、通知、系统回收和重启恢复。
- 强制停止结果只记录平台行为，不把不能被唤醒判为应用 bug。

### 15.3 每个 Task 的验证基线

1. 对应新增自动测试全部通过。
2. 完整 Hypium 回归通过。
3. API 24 `assembleHap` 通过，未引入 API 21 以上专属能力。
4. `git diff --check` 通过。
5. 更新 `docs/worklogs/S03-工作记录.md` 后，按 `S03-T<M>: <做了什么>` 提交并 push。

## 16. 主要风险与应对

| 风险 | 早期信号 | 应对 |
|---|---|---|
| 受限权限无法进入可交付 Profile | Task 3.0 安装、授权或审核失败 | 立即定级 L0，停止自动队列的生产接入，不修改签名模板伪造通过 |
| 后台回调不稳定 | 后台驻留 3 次测试出现遗漏 | 定级 L1，以启动/回前台补扫为正式体验 |
| 截图来源字段不稳定 | 不同截图方式或系统语言下字段不一致 | 使用多证据白名单；无法高置信则 L0，不上传普通图片 |
| 重叠扫描造成重复 | 同一媒体多次出现在候选中 | 队列与卡片双层 `sourceMediaKey` 唯一约束 |
| 崩溃窗口造成重复模型调用或通知 | `CARD_SAVED` 前后恢复测试失败 | 显式中间状态、稳定通知 ID、卡片来源幂等检查 |
| 自动上传引发隐私误解 | 用户不知道截图会发给外部模型 | 默认关闭、启用前说明、随时关闭；关闭后不再启动新请求并尽力取消在途请求 |
| Story 8 与本 Story 再次交叉 | Story 3 开始实现退避定时器 | 以第 13 节边界为准，退避和网络恢复全部留给 Story 8 |
