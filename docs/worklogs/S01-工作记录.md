### Task 1.2 配置模型与持久化 — [完成 2026-07-21]

- 实际做法（与计划的差异）：
  - 定义配置草稿、运行期配置、已验证元数据、加密信封与 `UNCONFIGURED / VERIFIED / CORRUPTED` 三态模型。
  - HUKS 使用不可导出的 AES-256 密钥和 AES-GCM/NoPadding；每次加密生成 12 字节随机 nonce，完整 HUKS 密文（含末尾 16 字节认证标签）以 Base64 存入 Preferences。
  - Preferences 使用 A/B 双槽和 `activeSlot` 指针；候选槽回读、解析并解密成功后才激活，激活失败时恢复旧指针。
  - `ConfigRepository` 只暴露非敏感元数据和受控临时解密回调，回调结束即清空持有的 Key 字符串引用。
- 遇到的问题与解决：
  - 当前工作区所在文件系统不支持 OHPM 创建依赖符号链接，Hypium 包下载完成后在 link 阶段报错；本机测试临时从 OHPM 缓存复制依赖，未把依赖目录加入版本库。换到 DevEco 默认支持的 NTFS 工作区后应重新执行 `ohpm install`。
- 验证方式与结果：
  - Hypium mock：首次保存与重启读取、密文落盘、加密失败保留旧槽、激活失败回滚、损坏 JSON 状态识别均通过。
  - `hvigorw test --mode module -p product=default`：通过；检查输出无测试 `ERROR`。
  - API 24 SDK 下 `assembleHap`：通过，生产 HUKS/Preferences 实现已参与 ArkTS 编译。
  - 尚未在模拟器或真机验证 HUKS 持久化；真机安全能力结果保留为待验证。

### Task 1.4 视觉验证门禁 — [完成 2026-07-21]

- 实际做法（与计划的差异）：
  - URL 预检使用 HarmonyOS 标准 URL 解析语义，拒绝带账号、查询参数或片段的地址；远程 HTTP 在请求前拒绝，仅允许主动确认后的 `localhost`、`127.0.0.1`、`::1` 回环例外。
  - 内置两张 64×64 PNG 颜色/形状关系图并随机选择；提示只给出判定规则和候选短答案，不直接透露图中实际关系。
  - 构造 OpenAI Chat Completions 视觉消息，使用 Base64 Data URL、`temperature=0`、12 token 上限和 15 秒连接/读取超时；HTTP 传输通过端口抽象，可由 fake transport 完整替换。
  - 响应兼容字符串 `content` 和文本数组 `content`，严格校验 `choices/message/content` 与挑战答案；认证、地址、视觉、网络、服务、响应格式和信任错误均使用固定脱敏文案。
  - `ConfigVerificationCoordinator` 只在视觉门禁成功后调用 `replaceVerified`；页面失败时只结束 loading 并显示分类错误，不改动三项草稿。编辑态未填写新 Key 时，通过 `withVerifiedConfig` 在受控解密回调内完成验证和事务式保存。
  - 自定义服务的已验证信任源随配置元数据恢复；仅当 `scheme + host + port` 变化时重置确认，路径编辑不要求重复确认。
- 遇到的问题与解决：
  - Hypium Windows 运行器没有 `@ohos.url` 原生实现；保留生产环境的 HarmonyOS 标准 URL 适配器，并为策略层增加可注入解析端口，使宿主测试可以覆盖 HTTPS、回环与自定义源门禁，不把宿主结果冒充设备集成验证。
- 验证方式与结果：
  - Hypium 共 24 项通过（0 Failure / 0 Error），其中视觉门禁 fake transport 覆盖 401、403、404、图片拒绝、错误答案、超时、5xx、非法 JSON、缺失 `choices/message/content`、字符串响应、文本数组响应、随机挑战、远程 HTTP、自定义源信任、回环 HTTP，以及成功后才写入。
  - API 24 SDK 下运行 `assembleHap`：通过，生产 HTTP 传输、标准 URL 适配器和页面协调逻辑均参与 ArkTS 编译。
  - `git diff --check`：通过；已检查本 Task 中文 diff，无编码损坏。
  - 未使用真实厂商凭证进行网络验证，也未在模拟器/真机验证 HTTP、HUKS 与交互流程；这些结果明确保留为待验证。
