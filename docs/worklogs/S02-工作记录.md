### Task 2.1 相册选图 + 端侧 OCR — [实现完成，真机验收待补]

- 实际做法（与计划的差异）：使用 `PhotoViewPicker` 单选图片；取消选择返回 `CANCELLED`。选中图片先通过只读文件描述符复制到 `filesDir/imports`，按识别到的 MIME 类型保留安全后缀，再从应用私有副本创建原始质量 `PixelMap` 并调用 Core Vision Kit `recognizeText()`。OCR 只统一换行并清理首尾空白；空结果或异常返回 `READY + RETRY_PENDING`。主页加入阶段性导入、处理中状态、私有副本信息和 OCR 预览，不含正式卡片。
- 发送确认：首次导入前按已验证配置的 `trustedOrigin` 说明 OCR 文本和缩略图的数据范围；确认记录存放于独立、非敏感 Preferences。服务源变化后必须重新确认。该步骤不读取或持有 API Key。
- 测试样例：在 `testdata/s02` 提交四张脚本确定性生成、无隐私的中文 PNG（讲座海报、取件码、DDL 通知、优惠券）及 `expected.json` 真值清单。
- 遇到的问题与解决：PhotoPicker 返回的媒体 URI 不能直接按普通路径复制；改为先 `fileIo.open()` 获取只读 fd，再由 fd 解码和复制。模拟器首次选择还会进入系统“仅可访问所选图片”确认页，这是系统隐私流程，不申请全相册读取权限。
- 自动验证：Hypium 36/36 通过；API 24 `assembleHap` 通过；`git diff --check` 通过。单元测试覆盖取消、成功、空 OCR、OCR 异常、文本规范化及资源释放。
- 模拟器验证：四张样例成功进入系统 PhotoPicker；讲座海报以 1080 × 1600、72158 bytes 完成应用私有复制。Core Vision 在当前 HarmonyOS Emulator 返回空文本，应用正确显示“待重试”且未崩溃。依既定口径，这不是 OCR 成功验收，四类 OCR 主要文字仍需 HarmonyOS NEXT 真机逐张复验。
- 对应 commit：本 Task 独立提交（见 Git 历史）。
