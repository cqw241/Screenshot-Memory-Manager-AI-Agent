## 〇、开发流程约定(每个 Story 必须遵守)

### 1. Story 启动

- 从最新 `main` 拉出新分支,命名:`feature/s<NN>-<slug>`(例:`feature/s01-config-wizard`)。
- 在分支上立即创建工作记录文档:`docs/worklogs/S<NN>-工作记录.md`,使用附录 A 模板,首次提交即包含该文件。

### 2. Task 完成

每完成一个 Task,必须依次执行:

1. 自测通过该 Task 的全部验收标准;
2. 更新 `docs/worklogs/S<NN>-工作记录.md`:填写该 Task 的完成日期、实际做法(与计划的差异)、遇到的问题与解决方式、验证方式与结果;
3. `git commit`(commit message 格式:`S<NN>-T<M>: <做了什么>`)并 `git push`。

**一个 Task 至少对应一次 push。** 未更新工作记录的 push 视为 Task 未完成。

### 3. Story 收尾

1. 核对 Story 验收标准全部满足;
2. 撰写交接文档:`docs/handover/S<NN>-交接文档.md`,使用附录 B 模板。**标准:默认交接者对本仓库和本项目毫无了解**,读完交接文档即可独立运行、验证、并在此基础上继续开发,无需口头答疑;
3. 发起 PR:`feature/s<NN>-*` → `main`。PR 描述包含:Story 验收标准逐条勾选、工作记录与交接文档的链接、演示截图或录屏;
4. PR 合并后,被本 Story 阻塞的 Story 方可启动。

### 4. Definition of Done(所有 Story 通用)

- 真机或模拟器实测通过(涉及后台任务/围栏/通知的必须真机);
- 无阻断性 bug,错误路径不崩溃;
- 工作记录完整覆盖每个 Task;交接文档达到"零上下文可接手"标准;
- PR 通过 review 合并进 `main`。

## Story 2:核心曳光弹 — 手动导入截图 → OCR → LLM 结构化 → 时间轴出卡

- **分支:** `feature/s02-tracer-pipeline`
- **阻塞于:** S1
- **交付物:** App 内手动选一张截图,贯穿 OCR → 多模态理解 → 落库 → 时间轴出卡的完整管线。这是全项目的主动脉,后续 Story 都在其上生长。

**Story 验收标准:**

- [ ] 手动导入讲座海报截图,时间轴出现类型、时间、地点正确的卡片,附缩略图
- [ ] 4 类目标截图(海报/取件码/DDL 通知/优惠券)各导入一张,类型识别正确
- [ ] LLM 返回非法 JSON 自动重试一次,再失败落"未分类"卡片,不崩溃
- [ ] 卡片持久化,重启后时间轴仍在

### Task 2.1 相册选图 + 端侧 OCR

photoAccessHelper 拉起相册选择器;选中图片走 Core Vision Kit OCR,输出全文文本。

- [ ] 对 4 类样例截图 OCR 出主要文字(取件码、日期、地点可见于输出)
- [ ] OCR 失败(如纯图形图片)不崩溃,返回空文本并标记待重试

### Task 2.2 LLM 理解客户端

封装解析调用:输入 OCR 文本 + 压缩缩略图(长边 ≤1024px),prompt 要求按固定 JSON schema 输出 `{类型, 实体{时间/地点/码号/金额}, 动作}`;非法 JSON 重试一次。

- [ ] 4 类样例均返回可解析的合法 JSON
- [ ] 人为注入非法响应(mock)时,重试一次后落"未分类",流程不中断
- [ ] 请求体中图片为缩略图而非原图(抓包或日志确认体积)

### Task 2.3 卡片存储层

RelationalStore 建卡片表(id、图片 URI、类型、实体 JSON、状态、创建时间、事件时间),封装增删改查与"按时间轴排序查询"。

- [ ] 插入/查询/更新/删除单测或手测通过
- [ ] 重启后数据完整;删除卡片不删相册原图

### Task 2.4 时间轴 UI

主界面时间轴列表:按事件时间倒序展示卡片(类型徽标、标题、关键实体、缩略图);点击进详情页看原图与全部字段。详情页提供删除入口与二次确认,删除后返回并刷新时间轴;只删除拾光卡片记录,不删除相册原图。

- [ ] 卡片信息与库中数据一致;空态有引导文案("试试导入一张截图")
- [ ] 详情页可查看原图
- [ ] 详情页删除需二次确认;删除成功后时间轴刷新,相册原图不受影响

### Task 2.5 端到端联调

把 2.1–2.4 串成"导入即出卡"的完整流,处理中有进度反馈。

- [ ] 从点击导入到出卡 ≤15 秒(正常网络),期间 UI 不假死
- [ ] Story 验收标准全部通过
