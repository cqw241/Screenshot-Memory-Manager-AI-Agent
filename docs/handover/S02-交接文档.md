# S02 核心曳光弹 · Story 2 → Story 3 交接文档

## 1. 项目是什么

“拾光”是一款 HarmonyOS NEXT 截图记忆管家 Agent，目标是把截图从相册中的静态图片转化为可理解、可回望、后续可主动触发的本地卡片。Story 2 建成手动摄取主链路：用户选图后，应用完成私有复制、端侧 OCR、多模态结构化理解、加密本地落库和时间轴展示。Story 3 将复用这条主链路，把入口从手动选图扩展为截图自动监听、持久化解析队列、完成通知与纠错。

依赖关系为 `S1 → S2 → S3 → S8/S9`。当前 Story 2 的代码和模拟器基线足以支持 Story 3 开发，但真机 OCR 仍没有验收证据。

> 当前状态：**Task 2.5 代码及模拟器联调完成，真机 OCR 待验。** Story 2 总验收尚未通过。

## 2. Story 2 交付了什么

- 从系统 PhotoPicker 单选图片；取消选择安全返回，不创建卡片。
- 将选中图片复制到应用私有目录，再从私有副本创建 `PixelMap` 并调用 Core Vision Kit OCR；OCR 空结果或异常进入 `RETRY_PENDING`，不阻断后续图片理解。
- 生成最长边不超过 1024 px、JPEG 85 的私有缩略图，并与 OCR 文本一起发送给设备内已验证的 OpenAI-compatible 多模态模型配置。
- 严格解析固定 JSON Schema；首次非法响应使用同一输入纠错一次，第二次仍非法则生成安全的“未分类”卡片。认证、网络或服务错误不会落库。
- 对日期证据做客户端校验：`eventTime` 非空时，`timeText` 必须明确包含年、月、日、时、分并与 ISO 组件一致；缺失年份的“7月23日前”等表达保留原文但令 `eventTime = null`。
- 使用加密 RelationalStore 持久化卡片，按事件时间构建时间轴；支持详情页查看私有原图和全部字段，并在二次确认后仅删除卡片记录。
- 使用 `ScreenshotIngestionCoordinator` 串联导入、理解和卡片写入，提供阶段反馈、并发防重、脱敏失败结果与端到端耗时。

本次未提交模拟器录屏。可复现的四样例、预期实体和生成方式位于 `testdata/s02/`，逐张实测结果记录在 `docs/worklogs/S02-工作记录.md`。

### 3.3 模拟器手工验收

1. 在 DevEco Studio Device Manager 启动 Pura 90 HarmonyOS 6.1.1（API 24）模拟器，安装并运行 `entry` 模块。

4. 在拾光中逐张选择并确认导入，检查阶段提示、缩略图、时间轴摘要和详情字段。
5. 预期讲座海报为 `POSTER`，时间 2026-07-25 14:00，地点“图书馆报告厅”；取件码为 `PICKUP_CODE`，码号 A8B6，地点“南门快递站”，`timeText` 保留“7月23日前”且 `eventTime = null`；DDL 为 `DDL_NOTICE`，时间 2026-07-28 23:59；优惠券为 `COUPON`，时间 2026-07-31 23:59，金额“满30减10元”。

## 4. 代码导览

### 4.1 目录与职责

- `entry/src/main/ets/pages/Index.ets`：启动门禁、手动导入入口、阶段反馈、耗时统计和时间轴刷新。
- `entry/src/main/ets/pages/CardDetail.ets`：按 `cardId` 读取详情、展示私有原图和字段、二次确认删除。
- `entry/src/main/ets/services/ScreenshotIngestionCoordinator.ets`：当前单图摄取协调器；串联三个窄接口并拒绝并发触发。
- `entry/src/main/ets/services/ScreenshotIngestionPorts.ets`：`ScreenshotImporter`、`ScreenshotUnderstander`、`ScreenshotCardWriter` 三个可替换端口。
- `entry/src/main/ets/services/ScreenshotImportService.ets`：PhotoPicker、私有复制和 OCR 的手动导入实现。
- `entry/src/main/ets/services/ScreenshotUnderstandingService.ets` 与 `StructuredResponseParser.ets`：缩略图请求、一次纠错、错误分类和严格结构/日期校验。
- `entry/src/main/ets/services/ScreenshotCardRepository.ets` 与 `HarmonyCardRowStore.ets`：卡片领域接口、数据校验和 RelationalStore 适配器。
- `entry/src/main/ets/models/ScreenshotCardPresentation.ets`：时间轴分组、标题、徽标和实体摘要的纯展示逻辑。
- `entry/src/test/`：宿主 Hypium 测试；`ScreenshotIngestionCoordinator.test.ets` 是 Story 3 复用主链路时最直接的契约示例。
- `testdata/s02/`：四张确定性无隐私样例和 `expected.json` 真值表。

### 4.2 关键数据流

手动入口的数据流为：

```text
Index
  → ScreenshotIngestionCoordinator
  → ScreenshotImportService
      → PhotoViewPicker → 私有原图 → Core Vision OCR
  → ScreenshotUnderstandingService
      → 私有缩略图 → 已验证模型配置 → 严格解析/一次纠错
  → ScreenshotCardRepository
      → 加密 RelationalStore
  → Index 重新读取 listTimeline() → 时间轴可见
```

协调器当前以“打开 PhotoPicker 的单图手动导入”为入口。Story 3 不应让媒体库监听器直接依赖页面或伪造点击流程，而应为“已知媒体 URI 的自动摄取”增加适配器/入口，再复用理解与卡片写入端口。Story 2 没有实现持久化队列、后台恢复、批量导入、通知或纠错编辑，这些属于 Story 3。

## 5. 已知问题与验收边界

- **真机 OCR 未验证。** 当前唯一设备是 Pura 90 API 24 模拟器，Core Vision 对四样例均返回空文本。不得从缩略图多模态识别成功反推 OCR 质量已经通过。
- **Story 3 开发时必须避免把真机 OCR 当作已验证能力。** 如果 Story 3 要基于 OCR 质量做产品判断，优先通过 `ScreenshotImporter` / `OcrSession` 等接口抽象注入确定性 OCR 文本、空文本和异常结果，以模拟数据推进；真机 OCR 只作为后续补验项。
- **后台能力的模拟器结果不是最终证据。** Story 3 涉及媒体库变更监听、后台驻留、通知和权限，这些按通用 DoD 仍需要真机验证。模拟器可以验证代码路径、队列状态机、去重和前台集成，但不能关闭对应真机验收项。
- **当前没有持久化解析队列。** 认证失败暂停、429/5xx 退避、杀进程续跑和多图去重尚未实现；不要把 `ScreenshotIngestionCoordinator` 的并发防重当作队列。
- **删除卡片不删除私有图片。** 这是 Story 2 的既定安全边界；Story 3 若引入清理策略，需要单独设计并明确不会影响相册源图。
- **HAP 未提交签名配置。** 使用本机 DevEco 开发者签名，禁止把签名材料加入仓库。
- **模型调用依赖外部服务。** 接口兼容性、限流和网络状态可能变化；自动测试应使用端口 fake，设备联调只记录脱敏分类和耗时。

## 6. Story 3 接手建议

1. 在 Story 2 PR 合并前需要并行开发时，从 `feature/s02-tracer-pipeline` 创建 `feature/s03-silent-ingest`；合并后再以 `main` 为基线同步，避免丢失 S2 管线。
2. 先为“已知媒体 URI → 私有复制/OCR 结果”的自动导入路径定义窄接口与确定性 fake，再接媒体库变更监听。不要让监听器持有页面状态、模型 Key 或数据库实现。
3. 将去重键、串行队列、任务状态和重启恢复做成独立持久化模块。先用模拟数据验证连续三张、重复事件、失败重试和杀进程恢复，再接生产监听。
4. 复用 `ScreenshotUnderstander` 和 `ScreenshotCardWriter`，统一成功/失败分类；保留 OCR 空结果仍可理解、模型服务错误不落库、二次非法响应保存未分类卡片的 Story 2 语义。
5. 通知与纠错以 `cardId` 为深链契约；纠错保存应复用 `ScreenshotCardRepository.updateUnderstanding()`，并确认时间轴和后续触发读取最新数据。
6. 权限拒绝必须降级为现有手动导入入口。模拟器上完成 UI、状态机和设置跳转后，把前后台监听、截图筛选、通知投递和授权后自动恢复明确列为真机补验。

建议 Story 3 首轮自动化至少覆盖：仅截图事件入队、拍照/普通保存忽略、重复媒体去重、串行处理、OCR 文本/空文本/异常三态、认证错误暂停、可重试错误保留任务、重启恢复、每张成功只发一次通知、权限拒绝降级。所有涉及 OCR 内容质量的断言都应来自确定性 fake，不得引用当前模拟器的空 OCR 结果作为通过依据。
