# S01 项目脚手架 + 首启配置向导 + 验证门禁 · 交接文档

## 1. 项目是什么

“拾光”是一款面向 HarmonyOS NEXT 的截图记忆管理应用，目标是把散落的截图整理成可检索、可回望的时间轴。Story 1 建立整个产品的安全入口：用户必须自行配置并验证一个具备图像理解能力的 OpenAI-compatible 模型，应用才会进入主界面。它位于依赖链起点 `S1 → S2 → S3/S4/S6/S7`，后续 OCR、结构化理解、时间轴出卡与触发能力都以本 Story 的已验证配置为前提。

## 2. 本 Story 交付了什么

- 建立 HarmonyOS NEXT Stage/ArkTS 工程，应用名“拾光”，Bundle Name 为 `com.shiguang.memory`，目标与最低兼容 API 均为 20。
- 提供三项模型配置向导：完整端点模板、可编辑模型建议、密码输入、高级套餐条款、自定义源信任和回环 HTTP 警告。
- 使用两个随机视觉挑战与 OpenAI Chat Completions 图像消息验证 `baseURL + apiKey + model`；认证、地址、视觉、网络、服务和响应格式错误均分类呈现。
- 使用 HUKS AES-256-GCM 加密 Key，并用 Preferences A/B 双槽事务式替换；验证失败不会落盘或覆盖旧配置，已保存 Key 不回显。
- 启动时异步解密检查配置：未配置进入向导，完整可解密配置进入空时间轴，损坏或版本不兼容进入带说明的恢复向导。
- 空时间轴提供后续可扩展的底部导航占位和模型设置入口；设置验证成功替换配置并回主页，失败保留草稿与旧配置，取消不写仓库并回主页。

Story 1 已于 2026-07-21 通过验收：HarmonyOS Emulator 完成全新首启、真实阿里云百炼视觉挑战、HUKS 加密保存、进入空时间轴，以及杀进程重启后的持久化读取。真实 HarmonyOS NEXT 真机仍属于后续兼容性复验范围。

### 3.3 手工验收步骤

1. 清除应用数据后启动：加载页结束后应进入配置向导，不能绕过向导进入时间轴。
2. 输入一个有效的 HTTPS 服务地址、API Key 和视觉模型；自定义服务需勾选信任。验证成功后应进入空时间轴。
3. 杀进程并重新启动：HUKS 解密与配置校验成功后应直接进入空时间轴。
4. 从右上角“模型设置”进入编辑页：Base URL 和模型会回填，Key 只显示不回显明文的占位提示。
5. 输入错误 Key 并验证：应停留在设置页、保留草稿并显示“认证失败”；取消后回到主页，旧配置仍可使用。
6. 再次进入设置，用有效候选配置通过挑战：新配置应事务式替换旧配置并自动回主页。
7. 用远程 `http://`、404 路径、纯文本模型/错误答案、断网或超时分别验证错误分类；所有失败均不得覆盖已验证配置。
8. 设备调试时人为破坏 Preferences 活跃槽或制造 HUKS 解密失败：重启不应崩溃，应进入带“需要重新建立模型连接”说明的恢复向导，并要求重新输入 Key。

## 4. 代码导览

### 4.1 目录与职责

- `entry/src/main/ets/pages/Index.ets`：根页面和四态 UI（加载、首次/恢复向导、空时间轴、设置修改）。
- `entry/src/main/ets/models/AppFlow.ets`、`services/StartupRouter.ets`：可测试的启动路由模型与三态决策。
- `models/EndpointCatalog.ets`：官方端点、编程套餐端点、推荐模型和服务源识别。
- `models/ModelConfig.ets`：配置三态、草稿、运行期配置、非敏感元数据和加密信封。
- `services/ConfigRepository.ets`：HUKS + Preferences 之上的受控读取与 A/B 双槽事务式替换。
- `services/HuksSecretCipher.ets`、`HuksErrorPolicy.ets`、`PreferencesKeyValueStore.ets`：HarmonyOS 生产加密、首次缺失密钥判定和键值存储适配器。
- `services/UrlPreflight.ets`：标准 URL 解析、HTTPS/回环门禁与请求地址规范化。
- `services/VisionChallenges.ets`、`VisionVerificationGate.ets`：本地图像挑战、请求构造、响应判定与错误分类。
- `services/HarmonyHttpTransport.ets`：可替换的 HTTP 生产传输。
- `services/ConfigVerificationCoordinator.ets`：先验证、后事务式保存的唯一协调入口。
- `entry/src/test/*.test.ets`：ConfigRepository、StartupRouter 和视觉门禁的宿主 mock 测试。

### 4.2 关键数据流

启动时 `Index → StartupRouter → ConfigRepository.getState()`；只有活跃信封完整、版本正确且 HUKS 能解密 Key 才返回 `VERIFIED` 并进入主页。首次配置或设置修改时，页面形成内存候选配置，经 `UrlPreflight → VisionVerificationGate → HttpTransport` 完成挑战；成功后 `ConfigVerificationCoordinator → ConfigRepository.replaceVerified()` 写入非活跃槽、回读解密验证，再切换活跃指针。任何预检、网络、解析、答案或写入失败都不会提前切换活跃槽。

重要决策详见 `docs/superpowers/specs/2026-07-20-story-1-config-wizard-decisions.md`，尤其是配置状态、服务信任、Key 安全、视觉验证和启动路由章节。

## 5. 已知问题与坑

- Story 1 已通过模拟器设备验收；真实 HarmonyOS NEXT 真机上的 HUKS、签名安装、网络权限和交互体验尚未复验，不应把模拟器结论外推为全设备兼容。
- HAP 当前未配置签名；请用本机开发者证书配置 DevEco 签名，不要把证书、口令或签名配置提交到版本库。
- D 盘卷异常会导致目录联接失败并使 OHPM 报 `UnNTFS`；继续开发应使用当前 C 盘健康副本，或先备份并修复 D 盘，不要依赖手工复制作为长期方案。
- 真实 OpenAI-compatible 服务对图像内容数组、错误体和模型 ID 的支持会变化；推荐模型仅是建议，最终仍以视觉挑战为准。
- 当前内置挑战用于能力门禁，不是防对抗认证协议；后续若扩大安全威胁模型，应增加挑战池或服务端下发挑战。
- 构建仍有 HUKS/Preferences 的异常处理警告和 `getContext` 弃用警告，均源自现有实现，应在不改变 API 20 兼容性的前提下另行清理。
